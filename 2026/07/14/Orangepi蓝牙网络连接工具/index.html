Orangepi蓝牙网络连接工具

前言

用了很长时间的orangepi，但每次使用都必须接屏幕，接一套键盘鼠标，非常麻烦，今天突发奇想，想到可以通过蓝牙来设置WIFI，毕竟之前接触过类似的设备，比如随身WIFI，云卓的遥控。更何况现在有了AI神力以后，直接让AI接入SSH来进行调试，力大飞砖。

做好后也是传到了仓库里，便于以后取用。

程序设计

一、设计原理

1.1 目标

香橙派5MAX开机后，用户用手机蓝牙搜索设备并连接，通过蓝牙串口（SPP）发送文本命令，实现：

  • 查看当前WiFi状态和IP地址
  • 扫描附近WiFi网络
  • 切换/连接WiFi（支持SSID或编号选择）
  • 断开WiFi

1.2 技术选型

环节 技术 说明
物理层 经典蓝牙 BR/EDR 板载UART蓝牙 hci1（SDIO蓝牙 hci0 不可用）
配对 SSP Just Works NoInputNoOutput模式，无需PIN码
传输 SPP（串口） RFCOMM通道，手机端用蓝牙串口终端App
SDP注册 BlueZ D-Bus ProfileManager1 绕过hci1损坏的SDP服务器
WiFi操作 NetworkManager nmcli 扫描、连接、状态查询
服务管理 systemd 两个独立服务，开机自启

1.3 为什么是经典蓝牙而非BLE

  • 市面上蓝牙串口App百花齐放，用户不需要安装特定App
  • SPP协议简单：手机发文本，板子回文本，和SSH一样自然
  • BLE GATT需要专门的App（nRF Connect等），用户门槛更高
  • 经典蓝牙的功耗问题在插电场景下可忽略

四、问题记录与解决方案

# 问题 原因 解决方案
1 bluetoothctl agent 执行后立即退出 单次命令模式exit，需交互模式保持stdin 放弃bluetoothctl，改用Python D-Bus Agent
2 Agent注册成功但配对请求不到达 GLib mainloop放在daemon线程内不调度 主线程运行GLib mainloop，SPP放后台线程/独立服务
3 手机提示 “PIN码不正确” Agent的RequestPinCode返回正确，但SSP与Legacy模式混用 启用SSP + NoInputNoOutput = Just Works
4 配对后连接提示 “read ret -1” hci1的SDP服务器完全损坏（sdptool browse local超时，内核报codec错误） 改用BlueZ D-Bus ProfileManager1注册SPP，附带完整SDP XML记录
5 SDP记录仍不生效 XML格式错误：RFCOMM协议描述符缺少UUID 0x0003 修复ProtocolDescriptorList为L2CAP+RFCOMM标准格式
6 服务进程退出不干净 非daemon线程阻塞进程退出 两个独立systemd服务，KillMode默认
7 偶尔WiFi/BT启动失败 USB/HDMI等外设在WiFi SDIO初始化时抢电流 early-boot脚本提前给WiFi上电（见供电优化文档）

关键教训

  1. BlueZ 5 的 SDP 服务器在某些UART蓝牙适配器上不可用 — 不要依赖 sdptool 或 rfcomm，必须用 D-Bus ProfileManager1 API
  2. GLib mainloop 必须跑在主线程或可靠的非daemon线程 — 否则 D-Bus 回调不会被处理
  3. 独立服务优于单体服务 — 配对和SPP各自独立，一个出问题不影响另一个
  4. SDP XML 记录格式严格 — ProtocolDescriptorList 必须包含完整的 L2CAP→RFCOMM 协议栈

使用方法

启动优先级

香橙派5有个不大不小的通病，就是当外部挂在过多的时候，主板容易供电不足，而导致WIFI模块没法启动。整个板子只有20W的功率，被USB和HTMI分一点出去，就不够启动WIFI模块了，开机没启动，后面也不会重新启动。
尝试过用一个简单的方法解决这个问题，就是配置一个service，让WIFI模块优先通电启动，这样就能避免开机后无法联网的问题。

新建一个wifi-power-first.service文件，再用sudu权限把它粘贴到/etc/systemd/system/路径下。
orangepi@orangepi5max:~/WIFIfirst$ sudo cp wifi-power-first.service /etc/systemd/system/
