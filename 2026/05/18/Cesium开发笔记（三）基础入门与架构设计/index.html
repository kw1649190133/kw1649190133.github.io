更现代的开发逻辑

前端技术的发展变化很快，早已不是一个http和css的事，为了能够更方便的进行开发和测试，随即出现了各种组件、工具，核心的思路还是将程序模块化，将系统拆分成小功能，通过组合与复用，构成一个完整的程序。

Vue和Vite

Vue是组件化的开发架构，一个用于构建用户界面的 JavaScript 框架。在传统的网页里，改一个按钮可能牵一发而动全身。而在 Vue 中，页面被拆分成一个个独立的“组件”（比如一个导航栏、一个数据卡片、一个弹窗）。这些组件像乐高积木一样，可以独立开发、反复使用，最后拼装成一个完整的复杂应用。对这种需要展示大量数据面板（如飞行参数、地图控件）的项目非常友好。
Vite是Vue的作者开发的前端构建工具，在没有 Vite 之前，传统工具（如 Webpack）每次修改代码都要把所有文件重新打包一遍，项目越大越慢，保存后可能要等好几秒甚至几十秒才能在浏览器看到变化。而 Vite 利用了浏览器的原生能力，能做到“秒开”服务器，你刚按下保存键，浏览器里几乎是瞬间更新，完全感觉不到延迟。
Vue 3 + Vite 标准工程结构如下

  • 关注点分离：页面长什么样交给 views 和 components，数据怎么获取交给 api，复杂的业务逻辑抽离到 hooks。当你发现 Cesium 的轨迹画不出来时，你只需要去查相关的工具函数或 API 调用，不用在一团乱麻的代码里大海捞针。
  • 高复用性：比如你在多个页面都需要展示无人机的姿态仪表盘，你只需要在 components/business 里写好一次，其他地方随时引用即可。

AI做程序的弊端

目前的AI用来做一些简单的函数或者自身可以逻辑闭环的程序，成功率还是比较高的，只要程序的规模不大，修改和测试都相对容易，就算中间缺少了一些环节，AI都能够自己暴力破解过去。但只要代码规模一上去，或者功能一变复杂，涉及到架构设计，AI出错的概率就会暴增，并且最大的问题是难以修改和维护，一旦让AI自己改，那问题可能会越改越多。
之前比较成功的开发经历，都是使用了一个非常简单的架构设计，就是线性分支，一个功能只对应一个子版块，子版块对应单独的py模块，模块之间互不干涉，最后用一个主界面程序进行调用。而这些模块都分别用AI初步开发，然后我再手动进行一些调整，尤其是最核心的那部分算法。
所以换到网页端程序也是如此，必须首先构建一个合适的文件结构，再去调整各种代码细节。

新的目录结构设计

重新梳理一下当前的目录结构：
node_modules 调用模块
public 静态资源
src
┗archive 历史垃圾堆文件夹
┗assets 图标文件夹
┗components
┗CesiumView.vue 主界面
┗VideoWindow.vue 视频读取界面
┗composables
┗useFlightPlayer.ts 数据清洗解析
┗services
┗CameraTrackManager.ts 相机跟踪管理器
┗DroneEnityManager.ts 无人机实体管理
┗FlightDataParser.ts 飞行数据解析器
┗types
┗drone.ts 接口定义
app.vue 程序主入口

现在文件输入路径：
文件输入 → FlightDataParser.parse() → DroneEntityManager.render() → Cesium，每一层只做自己分内的事，改任何一处都不会影响到其他层。
